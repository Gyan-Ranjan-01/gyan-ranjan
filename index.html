RepoMind Flagship

AI-Powered Repository Analysis Platform

View Source Try Live Demo

RepoMind enables developer teams to instantly map, explain, and query unfamiliar codebases recursively using Retrieval-Augmented Generation (RAG).

The Problem

Developers waste time reading massive files to grasp structure. Static search engines lack semantic coding dependencies and syntax contexts, slowing feature velocity.

The Solution

A semantic chunk-based RAG platform. Splits files into context-preserving chunks, embeds them in Qdrant via Gemini embeddings, and runs multi-query expansion for precise architectural answers.

Multi-Query RAG expansion via LangChain
Qdrant Vector similarity search
OAuth 2.0 Google auth + JWT sessions
Async FastAPI ingestion pipeline

* Query expansion handler

Ingestion: Repo ➔ FastAPI AST Chunker ➔ Qdrant DB • Query: Prompt ➔ FastAPI ➔ Qdrant Lookup ➔ Gemini LLM Response
Engineering Decisions & Trade-Offs
Decision Chosen Tech Alternatives Trade-offs & Rationale
AI Backend Engine FastAPI (Python) Node.js, Go
Trade-off: Higher container memory overhead than Go, but provides native integrations.
Benefits: Rapid interoperability with Python ML tooling (LangChain, Hugging Face, NumPy).
Limitations: Thread blocking during massive parser jobs; handled via Celery task delegation.
Vector Database Qdrant FAISS, Pgvector
Trade-off: Lacks SQL flexibility of Pgvector, but runs faster out-of-the-box.
Benefits: Native payload metadata filtering, gRPC speed, and persistent vector tracking.
Limitations: Requires managing an independent service instance.
Embeddings & Inference Gemini 1.5 Flash GPT-4o, Claude 3.5
Trade-off: Easing logic slightly behind Claude 3.5, but offers massive 1M+ token context windows.
Benefits: Drastically cheaper token processing for files indexing.
Limitations: Dynamic rate limits; managed with exponential client backoffs.
Key Challenge & Resolution

Standard chunking broke code semantics by splitting functions and classes. I implemented an AST-aware parser that preserved logical boundaries and structural metadata, significantly improving retrieval quality.

Lessons Learned

RAG quality depends on chunk preparation and structural metadata rather than raw LLM parameters. Syntax-aware databases improve relevance by 40%.

Future Outlook
  • Implement AST-based cross-file dependency graph indexing to resolve import networks.
  • Add support for real-time repository syncing using GitHub Webhooks.
  • Integrate local LLMs (Ollama) to allow offline codebase analysis.

DocOnCall

AI-Powered Healthcare Assistance Platform

View Source Try Live Demo

DocOnCall is a full-stack medical assistant platform that handles symptom-based AI chat, PDF report analysis, and keyword-driven specialist doctor recommendations.

The Problem

Patients struggle to understand medical PDFs, creating anxiety. Doctors spend valuable consulting time on standard, repetitive screening steps.

The Solution

An MVC Express platform using Gemini (primary) + Groq LLaMA-3 (fallback) for AI chat, Gemini Vision for PDF report parsing, and symptom keyword matching to route patients to specialist doctors.

11 Models Gemini + Groq fallback chain
Dual Portal Patient & Doctor dashboards
MVC Pattern Express + EJS architecture
Keyword AI Symptom → specialist routing
Architecture Blueprint
🖥️ Client UI (HTML/CSS) Symptom reporting forms & report uploads
🔐 Session Auth express-session with httpOnly cookies
🚀 Express Controllers Routes requests & sessions validation
🛡️ Redaction Pipe Regex redaction strips PII from reports
🍃 MongoDB Stores chat records and profiles
🧠 Groq / Gemini LLM processing (Llama-3 & Gemini APIs)
Workflow: Request ➔ Session Check ➔ PII Redaction ➔ Database / AI Inference ➔ Controller Response
Key Challenge & Resolution

Transferring medical reports directly to cloud models risked PII leaks. I implemented a local regex redaction layer that strips names and phone numbers before sending data to external APIs.

Built with: Node.js Express.js MongoDB Gemini API Groq Express-Session
